# Conversion-kernel baseline summary — 2026-04-23

Full-tier baseline at commit `fb92135` on AMD Ryzen 9 7950X, zenbench
interleaved round-robin. Each bench measures throughput (src + dst
bytes per second) via `RowConverter::convert_row`. Sizes: 256 px (L1),
4096 px (L1/L2), 1 920×1 080 (memory).

## Ranked by gap = (nearest-neighbor throughput) / (this throughput)

Only 4096 px shown — 256 px tracks it closely and 1080 p is memory-bound
and mostly confirms the L2 story.

### Tier A — huge gaps, wide impact (do first)

| Kernel | Throughput | Comparable path | Gap |
|---|---|---|---|
| U8→F16 (via F32) | **1.41 GiB/s** | U8→U16 at 112.82 GiB/s | **80×** |
| U16→F16 (via F32) | 1.86 GiB/s | U16→U8 at 34.39 GiB/s | 18× |
| F32→F16 | 3.25 GiB/s | F32→U16 at 64.33 GiB/s | 20× |
| F16→F32 | 7.09 GiB/s | U16→F32 at 88.68 GiB/s | 12× |
| Straight→Premul RGBA F16 | 0.99 GiB/s | RGBA F32 at 13.74 GiB/s | 14× |
| Premul→Straight RGBA F16 | 0.94 GiB/s | RGBA F32 at 7.51 GiB/s | 8× |
| Straight→Premul GrayA F16 | 2.20 GiB/s | GrayA F32 at 33.52 GiB/s | 15× |

**Root cause**: all F16 paths funnel through the local scalar
`f16_bits_to_f32` / `f32_to_f16_bits` in `src/f16_scalar.rs`. F16C SIMD
(stable since Rust 1.27, MSRV-safe at our 1.89) would vectorize these
8 lanes at a time — projected ~8× lift, closing most of the gap.

**Wide impact**: every zenpixels-convert consumer that touches F16
(zentiff FP16, Ultra HDR gain maps, HEIF/AVIF F16 intermediates, JXL
modular F16) is paying this cost today. A single SIMD wrap of the two
core functions fixes all 8+ F16 kernels at once.

### Tier B — large gaps, narrower impact

| Kernel | Throughput | Comparable path | Gap |
|---|---|---|---|
| Linear F32 → PQ U16 | **1.39 GiB/s** | sRGB U8→Linear at 24.26 GiB/s | 17× (vs best TF) |
| PQ U16 → Linear F32 | 2.68 GiB/s | | 9× |
| Linear F32 → sRGB U8 | 4.56 GiB/s | sRGB U8 → Linear at 24.26 | 5× (asymmetric!) |
| Linear F32 → PQ F32 | 2.72 GiB/s (T4) | Linear → sRGB F32 at 24.95 | 9× |
| Linear F32 → HLG U16 | 4.44 GiB/s | | — |

**Root cause**: [linear-srgb#10](https://github.com/imazen/linear-srgb/issues/10)
— BT.709/PQ/HLG/Gamma22 slice APIs are scalar in upstream. Their SIMD
rites exist but aren't wired to public slice dispatchers. Fix: upstream
PR, or wrap the rites locally via `incant!` while waiting.

**Narrower impact** because PQ/HLG are HDR-only; BT.709/Gamma22 are rare
TFs. But within HDR pipelines, this is 5–17× on every encode or decode.

### Tier C — asymmetric sRGB OETF vs EOTF

| Kernel | Throughput | Notes |
|---|---|---|
| sRGB U8 → Linear F32 | 24.26 GiB/s | Forward: 256-entry LUT, ~1-cycle lookup |
| Linear F32 → sRGB U8 | 4.56 GiB/s | Reverse: rational poly + divide per lane |
| sRGB U8 fused gamut | 3.79 GiB/s | Inherits the OETF cost |
| Linear F32 → sRGB U8 + gamut | 3.11 GiB/s | Inherits the OETF cost |

**Root cause**: rational polynomial with f32 divide. Divide is 15–20
cycles per lane on x86, and SIMD divide is not meaningfully parallelized
on Ryzen. Fix options:
1. Larger LUT + linear interpolation (512 / 1024 entries) — trades
   cache pressure for throughput.
2. Newton-Raphson reciprocal estimate instead of true divide. Cheaper
   per lane but needs one extra mul + fma.
3. Accept as-is if not in a hot path — but this affects every encode
   to u8 in an sRGB workflow, so it is in a hot path.

Upstream `linear-srgb` work.

### Tier D — off-limits by directive

| Kernel | Throughput | Notes |
|---|---|---|
| Linear RGB → Oklab | 1.61 GiB/s | Interleaved `fast_cbrt` per lane |
| Linear RGBA → Oklaba | 2.14 GiB/s | |

Per the "interleaved Oklab is useless" directive: no optimization
pass here. Planar-SIMD Oklab lives in zenfilters via
`magetypes::cbrt_lowp` and scatter/gather.

No action. These are doing what they should. Our refactored U16 alpha
kernels sit here too (Straight→Premul RGBA U16 at 22 GiB/s).

## Revised priority list

1. **F16C SIMD for `f16_bits_to_f32` / `f32_to_f16_bits`** — Tier A. One
   localised change (`src/f16_scalar.rs` + callers in convert_kernels.rs),
   wide impact. Projected 8× on ~8 kernels simultaneously. Stable Rust,
   MSRV-safe.
2. **Linear-srgb TF slice dispatch (issue #10)** — Tier B. Upstream PR
   covering BT.709, PQ, HLG, Gamma22 slice fns. Benefits every HDR
   pipeline 5–17×. Blocked on upstream accepting the scope.
3. **Investigate sRGB OETF asymmetry** — Tier C. Upstream. Either larger
   LUT or reciprocal-estimate divide.
4. **Matte_composite TF-aware fused kernel (previous spike target)** —
   reclassified as medium priority. Only ~1-5% usage; the promised 3–5×
   speedup on a narrow kernel moves less than #1 which is 8× on the
   common F16 path.
5. **Premul→Straight U16 divide** — Tier B-ish (2.17 GiB/s, ~10×
   behind comparable U8). Lower priority but noted.

## What's conspicuously absent

- **No gamut action needed.** The matrix itself is fine (22 GiB/s linear
  f32). The slow-gamut kernels are slow because of the TF at one end,
  not the matrix.
- **No Oklab action** per the planar-SIMD-is-zenfilters directive.
- **No depth-scaling integer action.** U8↔U16↔F32 is already at memory
  bandwidth via garb.

## Next step

Build the F16C SIMD kernel for `f16_bits_to_f32_x8` / `f32_to_f16_bits_x8`
as a pilot, validate correctness (exhaustive 65 536 roundtrip already
exists in `f16_scalar.rs` tests — extend it to cover the SIMD path),
bench against the scalar loop using bench_t2_depth. Expected: U8→F16
lifts from 1.4 GiB/s to ~10 GiB/s at 4096 px.
